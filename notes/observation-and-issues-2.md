# User Observations, issues and new features required

## Refinements

 1. The taxonomy should be displayed in the selected language of the hub. If there are related terms in another language, they should be optionally revealed by the user by clicking the edit or view other languages button. The user should be able to select another language to displaty its taxonomy, terms and descriptions in that selected language.
 2. When a user is in the publishing modal and has the flavors visible, it may be apparant the a flavor may not be expressed correctly for a native speaker of the language, for example if it is in Afikaans. If the user has edit rights, could be useful to be able to view the wording and to edit the label without navigating to the taxonomy page.
 3. When the user is in the publishing modal, and the user believes that another taxonomy flavor is also applicable, the user should be able to add the flavor to the publication. This can be done with a search and selection, where the user can search for the flavor and add it to the publication. This should show the description so that the user can confirm that the flavor is correct. It must still be possible to suppress a flavor that was auto assigned.
 4. Having alliases for the flavors might be useful in future. How could this be implemented?
 5. The left menu shows 'History'. A better term might be 'Publications' or 'Published'. 
 6. When a merge is performed on the taxonomy term, the descriptions should be combined if they differ so that both terms are clearly included. The user can edit as needed.

## Curator announcements, promos and commercials
Curators will have a need to insert content of a promotional nature. These could be in the form of:
 - Community announcements
 - Promotions (paid)
 - Commercials (paid)
 - Campaigns (with unique campaign codes)

These should all be handled in a similar way. They should have: 
 - a start date
 - an end date
 - paramaters that will determine how often the posting will inserted into the stream.
 - Whether the recipient (e.g. the user of the xentara reader, has the option to supress future views of these items in the stream.)

The xentara reader beahviour (still to be fukky defined in another project) will be interactive, i.e. users can suppress future views of these items in the stream. However Telegram channel feeds should be immutable unless the the user uses the telegram bot as a private feed.

The 'look' of these promotional items should be distinct from the regular articles. They should have links to the relative content. These could be web urls, hashtags or anything else the curator may desire.  

In addition, xentara studio's own invitation to users to sign up or join should be a recurring item in the publication stream but if the user is already a member it should not be displayed. There is no start or end date for these. This would probably require a xentara studio admin backend function.
